# `core.membership` — Tenancy (User → Org → Role)

> Part of the [Core module reference](README.md). Source: [`app/core/membership.py`](../../app/core/membership.py). See also: [auth & authorization architecture](../architecture/auth-and-authorization.md).

## Purpose & responsibilities

Owns the org tenancy model: creating orgs, adding/removing members, role
changes, and the idempotent user-provisioning hook Cognito's post-confirm
Lambda calls. This is the module every other Core/service authorization
check ultimately reads from.

## Internal architecture — single-table DynamoDB design

One table (`a2z-core-membership`) using the adjacency-list pattern:

```mermaid
erDiagram
    USER_ITEM {
        string PK "USER# prefix + sub"
        string SK "METADATA"
        string email
        string created_at
    }
    ORG_ITEM {
        string PK "ORG# prefix + org_id"
        string SK "METADATA"
        string name
        string owner_id
        string created_at
    }
    MEMBERSHIP_ITEM {
        string PK "USER# prefix + sub"
        string SK "ORG# prefix + org_id"
        string GSI1PK "ORG# prefix + org_id"
        string GSI1SK "USER# prefix + sub"
        string role
        string joined_at
    }
```

- **Base table key** (`PK`/`SK`) answers "what orgs is this user in" —
  `list_user_orgs` queries `PK = USER#{sub} AND begins_with(SK, "ORG#")`.
- **GSI1** inverts the key so `list_org_members` can answer "who is in this
  org" in one query: `GSI1PK = ORG#{org_id}`.
- User and org metadata live as sibling items (`SK = "METADATA"`) in the same
  table rather than separate tables — a deliberate single-table design so a
  user's or org's full picture is reachable via one partition.

## Public API

| Function | Signature | Notes |
|---|---|---|
| `get_membership` | `(user_id, org_id) -> Membership \| None` | Single `GetItem`. < 50ms target |
| `list_user_orgs` | `(user_id) -> list[Org]` | Query + per-org `GetItem` fan-out via `asyncio.gather`. < 100ms |
| `list_org_members` | `(org_id) -> list[Membership]` | GSI1 query, sorted owner-first | < 200ms |
| `create_org` | `(org_name, owner_id) -> Org` | `TransactWriteItems`: org record + owner membership, atomically |
| `add_member` | `(org_id, user_id, role, inviter_id) -> Membership` | Conditional put; raises `AlreadyExistsError` if already a member |
| `change_role` | `(org_id, user_id, new_role, changer_id) -> Membership` | Conditional update; raises `NotFoundError` if no membership |
| `remove_member` | `(org_id, user_id, remover_id) -> None` | Conditional delete; **caller must prevent removing the last owner** (not enforced here) |
| `create_user_if_not_exists` | `(user_id, email) -> None` | Idempotent; called by the Cognito post-confirm Lambda |

`org_id` is generated by `create_org` as `{slugified-name}-{uuid4 hex[:8]}` —
never client-supplied, so collisions and injection via org name are both
structurally avoided.

## Dependencies

`core.audit` (every mutation logs), `core.events` (`member.added`,
`member.removed`, `member.role_changed` are published on every mutation),
`core._ddb`, `core.clients`, `core.exceptions`.

## Data models (Pydantic)

```python
class Role(str, Enum):
    OWNER = "owner"; ADMIN = "admin"; MEMBER = "member"; GUEST = "guest"

class Membership(BaseModel):
    user_id: str; org_id: str; role: Role; joined_at: datetime

class Org(BaseModel):
    org_id: str; name: str; owner_id: str; created_at: datetime
```

`Role` is the vocabulary every authorization check in this repo interprets —
see [role-mapping gap](../architecture/auth-and-authorization.md#role-mapping-gap-documented-not-silently-resolved)
for how Omni-Channel maps its own Owner/Admin/Agent/Viewer language onto it.

## Error handling

| Error | Status | Raised when |
|---|---|---|
| `AlreadyExistsError` | 409 | `add_member` on an existing membership |
| `NotFoundError` | 404 | `change_role`/`remove_member` on a non-existent membership |
| `MembershipError` | 400 | Any other DynamoDB failure |

## Security considerations

- **Cross-org isolation is structural**: every read/write key includes
  either `PK = USER#{sub}` or `GSI1PK = ORG#{org_id}` — there is no query
  shape that returns another org's members or a user's membership in an org
  they don't belong to.
- **`remove_member` does not prevent removing the last owner.** This is a
  documented caller responsibility, not enforced in Core — a router/service
  building on this must add that check itself if it matters for its UX.
- Role changes and removals are logged to both `core.audit` and published
  as EventBridge events, so any privilege change is traceable two ways.

## Example usage

```python
from app.core import membership
from app.core.membership import Role

org = await membership.create_org("Acme Jewelry", owner_id="auth0|abc123")
m = await membership.add_member(org.org_id, "auth0|def456", Role.MEMBER, inviter_id="auth0|abc123")
members = await membership.list_org_members(org.org_id)  # owner-first
```

## Extension points

- New roles: add to the `Role` enum and `_ROLE_ORDER`; every caller that
  does `role in {...}` needs its own review — there is no central
  permission matrix to update (`CLAUDE.md` §14).
- Org metadata fields (beyond `name`/`owner_id`): add to `Org` and the item
  builder in `create_org` — additive DynamoDB changes need no migration
  (`docs/migrations.md`).

## Known limitations

- No pagination on `list_org_members` — every membership for an org comes
  back in one query. Fine at current scale; would need a paginator for a
  very large org.
- First-login org bootstrap (create a default org on a brand-new user's
  first request) is explicitly **not** implemented anywhere yet — see
  [auth & authorization](../architecture/auth-and-authorization.md).
